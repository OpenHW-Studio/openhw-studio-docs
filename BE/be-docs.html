OpenHW Studio Backend Documentation

This page documents the current Express + MongoDB backend found in openhw-studio-backend/src. It covers runtime setup, route groups, major controllers, data models, authentication flow, classroom APIs, uploads, gamification progress, compilation endpoints, and component-management utilities.

Express 4 API server with modular route mounts
MongoDB Mongoose models for users, classes, submissions, and progress
Passport Google OAuth plus JWT-protected application routes
Multer Classroom asset upload pipeline into frontend public assets

Runtime Overview

How the backend boots, what middleware it loads, and what it exposes.

Entry Point

src/server.js loads env values, ensures temp/component directories exist, connects to MongoDB, starts Express, and mounts /api and /auth.

Core Middleware
  • cors()
  • express.json({ limit: '25mb' })
  • express.urlencoded({ extended: true, limit: '25mb' })
  • express-session
  • passport.initialize() and passport.session()
Static Assets

/examples serves files from openhw-studio-examples/examples.

Start Scripts
  • npm run start: production-style node start with .env
  • npm run dev: nodemon watcher

Architecture Notes

Useful high-level behavior inferred from the backend modules.

Auth Strategy

Protected app APIs use JWT via Authorization: Bearer ... or the jwt cookie. Google OAuth is exposed separately under /auth.

Classroom Ownership Rules

Teacher-only mutations are enforced inside the classroom controller. Student access is verified against class membership.

Upload Storage

Classroom uploads do not store file bodies in MongoDB. The backend returns URL paths for uploaded files and the DB stores those string paths.

Coverage

Main domains are auth/users, classroom management, submissions, comments, compile/flash, library operations, component moderation, and progress/gamification.

Primary Data Models

High-value collections and the fields they are built around.

User

Identity model for student, teacher, and admin accounts.

  • name, email, password, googleId
  • role: student | teacher | admin
  • college, branch, semester, bio, image
  • classes, points, coins, level, badges

Class

Root classroom aggregate for teacher ownership and enrollment.

  • name, bio, image, joinCode
  • teacher reference
  • students, assignments, notices references

Assignment

Teacher-authored classwork items with due dates and resource files.

  • classId, title, description
  • templateProjectId, dueDate
  • attachments / files
  • createdBy

Notice

Teacher stream posts for a class.

  • classId, title, message
  • attachments / files
  • createdBy

Submission

Student assignment hand-ins, notes, and uploaded files.

  • assignmentId, studentId, classId
  • projectId, notes, attachments, files
  • score, feedback
  • Unique index on (assignmentId, studentId)

Comment

Discussion records attached to assignment or notice stream items.

  • classId, postId, postType
  • message, createdBy

Project

Simulation/project storage for boards, components, and source code.

  • userId, board
  • components, connections, code
  • isAssignment, assignmentId

Progress / UserProgress

Gamification state for XP, levels, unlocked components, quiz attempts, completed projects, badges, and streaks.

  • level, xp, coins, badges
  • quizAttempts, completedProjects, unlockedComponents
  • currentStreak, longestStreak, lastActiveDate

Implementation Notes

Important behavior inside controllers and middleware.

authMiddleware.js

Extracts JWT from either bearer header or jwt cookie, verifies it, fetches the user, and attaches req.user.

passport.js

Google OAuth strategy support for browser-based auth callback flow.

classroomUpload.js

Builds upload destinations and public paths for classroom/profile assets. Used with Multer for file ingestion.

classroomController.js

Central classroom domain controller. Contains teacher authorization checks, join-code enrollment, assignment lifecycle, notice lifecycle, comment lifecycle, submission reads/writes, and upload response shaping.

compileController.js

Handles Arduino compile, flash, and serial-port discovery flows. This is the hardware-facing part of the backend.

componentController.js

Implements component submission, approval, rejection, install listing, deletion, and backup/export for custom component workflows.

Suggested Reading Order

If someone is new to this backend, this order gets them productive fastest.

1. Boot Flow

Read src/server.js, then src/routes/api.js and src/routes/auth.js.

2. Auth + User Model

Read src/middleware/authMiddleware.js, src/controllers/userController.js, and src/models/User.js.

3. Classroom Domain

Read src/routes/classroom.js, then src/controllers/classroomController.js, then the classroom-related models.

4. Specialized Domains

Move on to compile, progress, libraries, and component moderation.
